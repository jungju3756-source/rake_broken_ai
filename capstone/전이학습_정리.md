## 2. 전이학습(Transfer Learning)이란 — 왜 쓰는가

전이학습 = **이미 대규모 데이터(COCO 80클래스, 수십만 장)로 학습된 모델의 가중치를 출발점으로** 삼아, 우리 데이터(제진기 레이크)로 **추가 학습(fine-tuning)** 하는 것.

- 장점: 적은 데이터(수십~수백 장)로도 빠르게 좋은 성능. 밑바닥부터 학습 대비 압도적으로 효율적.
- YOLOv8n 의 백본/특징추출기는 "선·모서리·질감" 같은 일반 시각 특징을 이미 잘 뽑음 → 우리는 "부서진 빗살" 같은 **도메인 특화 패턴만 추가로** 가르치면 됨.

> 실제로 COCO 사전학습 모델을 그대로 이 영상에 돌리면 거의 아무것도 검출 못 함(도메인 불일치). → 전이학습이 반드시 필요한 이유.

`03_train.py`에서 전이학습이 일어나는 핵심 한 줄:
```python
model = YOLO("yolov8n.pt")   # COCO 사전학습 가중치 로드 = 전이학습의 출발점
model.train(data="...data.yaml", epochs=100, ...)   # 우리 데이터로 fine-tuning
```
학습 로그의 `Transferred 319/355 items from pretrained weights` 가 "사전학습 가중치를 이어받았다"는 증거.

## 3. 단계별 수행 상세

### ① 프레임 추출 — `01_extract_frames.py`
동영상을 라벨링 가능한 정지 이미지로 변환.
```bash
python 01_extract_frames.py --video "../오류 동영상.mp4" --out frames/rake --prefix rake --every 12
```
- `--every 12` : 24fps에서 12프레임마다 1장(초당 2장) → 83초 영상에서 166장
- `--prefix rake` : **한글 영상명 회피용**. (아래 트러블슈팅 참고)

### ② 중복 제거 / 다양한 프레임 선별 — `04_select_diverse.py`
레이크 영상은 변화가 느려 **인접 프레임이 거의 동일** → 그대로 쓰면 과적합(아래 4장 참고).
프레임들의 축소 시그니처를 비교해 **서로 가장 다른 80장**만 탐욕적으로 선택.
```bash
python 04_select_diverse.py --video "../오류 동영상.mp4" --out frames/rake_v2 --target 80 --prefix rake
```

### ③ Roboflow 라벨링
- **정식 Project 생성**: Workspace → Create New Project → **Object Detection**
  (주의: 첫 화면의 "Build a Computer Vision Model in Minutes" 퀵스타트는 10장 제한 → 쓰지 말 것)
- `Upload Data` 로 80장 업로드
- **클래스 `rake_broken`** : 휘고 부서진 빗살 부위에만 **타이트한 박스**. 레이크 바 전체를 잡지 않음.
- 손상이 안 보이는(정상) 프레임은 박스 없이 두면 배경/정상 샘플로 사용.
- Train/Valid/Test = **70/20/10** (= 56/16/8) 으로 버전 생성 → **Export: YOLOv8** → zip 다운로드

### ④ 데이터셋 배치 + data.yaml 경로 수정
Roboflow가 만든 `data.yaml`은 경로가 `../train/images` 라 Ultralytics에서 어긋날 수 있음 → **절대경로로 고정**:
```yaml
path: C:/Users/aim88/Desktop/26_4/방학_캡스톤_디자인/rake-fault-v2.v1i.yolov8
train: train/images
val: valid/images
test: test/images
nc: 2
names: ['rake', 'rake_broken']
```

### ⑤ 전이학습 실행 — `03_train.py`
```bash
python 03_train.py --task det
```
주요 하이퍼파라미터(8GB VRAM 기준):

| 파라미터 | 값 | 의미 |
|---|---|---|
| base | yolov8n.pt | 사전학습 가중치(전이학습 출발점) |
| epochs | 100 | 최대 학습 횟수 |
| imgsz | 640 | 입력 해상도 |
| batch | 16 | 한 번에 학습하는 이미지 수(OOM시 8) |
| device | 0 | GPU |
| patience | 30 | 30에폭 개선 없으면 조기종료(EarlyStopping) |
| hsv_h/s/v, fliplr, mosaic | — | 조도/반사/위치 변화 대응 증강 |

- 학습 결과 가중치: `best.pt`(검증 최고 성능), `last.pt`(마지막 에폭)
- ⚠️ Ultralytics 기본 저장 경로(settings)가 다른 폴더로 잡혀 있어, 실제 저장 위치를 확인 후 `models/`로 복사함:
  `models/rake_broken_best.pt`

### ⑥ 영상=카메라 입력 추론 — `02_infer_video.py`
`cv2.VideoCapture` 는 **카메라 인덱스(0)와 동영상 경로를 동일 API로** 처리 → 코드 수정 없이 전환.
```bash
# 동영상(현장 카메라 대용)
python 02_infer_video.py --model models/rake_broken_best.pt --source "../오류 동영상.mp4" --conf 0.25 --save out_rake_broken.mp4
# 실제 USB/IP 카메라
python 02_infer_video.py --model models/rake_broken_best.pt --source 0 --show
```
- `--show` : 실시간 검출 창 (ESC 종료)
- `--conf` : 신뢰도 임계값(낮추면 많이/높이면 확실한 것만)
- 결과: 1992프레임을 약 **68 FPS** 로 처리(실시간 가능)

## 4. 실험 결과 — 1차 vs 2차 (중요한 교훈)

| 구분 | 1차 (whole rake) | 2차 (broken teeth) |
|---|---|---|
| 라벨 대상 | 레이크 **전체** | **부서진 빗살 부위만** |
| 클래스 | `rake` 1개 | `rake`, `rake_broken` 2개 |
| 프레임 | 166장(거의 중복) | 80장(중복 제거) |
| 분할 | 116/33/17 | 56/16/8 |
| mAP50 | **0.991** | rake_broken **0.456** |
| 평가 | ❌ **가짜 높은 수치** | ✅ **현실적 수치** |

### 왜 1차의 99%는 "좋은 게 아니라 경고"인가
166장이 거의 중복(인접 프레임) → train과 valid에 **거의 같은 그림**이 섞임(데이터 누수, data leakage) → 모델이 사실상 "외워서" 100%. 실제 현장 성능 아님.
→ 2차에서 **중복 제거 + 손상 부위 라벨**로 바꿔 현실적인 모델을 얻음.

### 2차 모델의 실제 검출(영상)
- 또렷한 손상: `rake_broken` conf **0.6+** 로 정확히 검출
- 멀리 있는 작은 손상: conf 0.3 안팎(약하게 검출) — 작은 객체라 해상도 한계
- 즉 **수치(mAP)는 낮아 보여도 시각적으로는 고장 부위를 잘 잡음**

## 5. 트러블슈팅 (실제 겪은 문제)

| 문제 | 원인 | 해결 |
|---|---|---|
| `cv2.imwrite`가 프레임을 저장 안 함 | **한글 경로/파일명**(Windows에서 imwrite 실패) | `cv2.imencode`+`tofile` 사용, 출력 파일명 ASCII(`--prefix`) |
| GPU 안 잡힘(`cuda available: False`) | pip 기본 torch가 **CPU판** | cu121 버전으로 재설치 |
| Roboflow "max 10 images" | 첫 화면 **퀵스타트 위젯** 제한 | 정식 **Project → Upload Data** 로 진입 |
| mAP 100% | 중복 프레임 데이터 누수 | 중복 제거(04 스크립트) |
| best.pt가 엉뚱한 폴더에 저장 | Ultralytics settings 기본 경로 | 실제 경로 확인 후 `models/`로 복사 |

## 6. 개선 방향 (다음 단계)

1. **`imgsz` 640 → 1280** 재학습 — 부서진 빗살은 작은 객체라 해상도↑가 mAP에 가장 효과적
   ```bash
   python 03_train.py --task det --imgsz 1280 --batch 8
   ```
2. **데이터 보강** — 손상 잘 보이는 구간에서 프레임 추가, 정상 빗살 프레임도 일부 포함(오탐 방지)
3. **운영 임계값 튜닝** — 배포 시 `--conf 0.4` 등으로 확실한 고장만 알림
4. **시간 정보 활용** — 단일 프레임보다 "N초간 손상 지속 = 고장 확정" 후처리로 오탐 감소
